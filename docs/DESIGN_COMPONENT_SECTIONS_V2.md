# Component Sections – Design v2 (single stored YAML)

**Status:** Design only. No coding.

---

## 1. Principles

- **Single source of truth:** One stored YAML document per device. Created from the device’s hardware recipe at creation (with substitutions). Everything else is derived or amends this document.
- **No dual model:** No separate “user additions” vs “recipe + compiler.” The stored YAML is the base; user edits replace whole sections; app/compiler amend it by replace, patch, or append as needed.
- **Component dialog** is a view over that stored YAML, split into sections, with clear state and actions.

---

## 2. Stored YAML and “original” for Reset

- When a new device is created, the stored YAML is populated from the **current** device hardware recipe (with substitutions: device name, API key, etc.).
- **Reset** (per section or Reset All) restores to **whatever the current hardware recipe is** at the time of Reset (with substitutions reapplied). So if the recipe file has changed since the device was created (e.g. integration update), Reset gives the user the updated recipe content. We do **not** store a snapshot of “original at creation”; we recompute “original” from the current recipe when needed.

---

## 3. Section states and UI

Each section in the Component dialog has a clear state:

| State    | Meaning |
|----------|--------|
| **Empty** | No content for this section in the stored YAML (or section absent). |
| **Auto**  | Section has content from the recipe/defaults and has **not** been user-saved. (Content matches current recipe for this section.) |
| **Edited** | User has saved changes for this section; stored content differs from current recipe for this section. |

- **Derivation:** We can derive Auto vs Edited by comparing the section’s stored content to the content produced by the **current** recipe (with substitutions) for that section. No separate “user edited” flag required.
- **Actions per section:** **Save** (write this section’s current text into the stored YAML; section becomes Edited unless it matches recipe), **Reset** (replace this section in the stored YAML with current recipe for this section; section becomes Auto or Empty).
- **Actions for the dialog:** **Save All** (persist all section contents from the dialog into the stored YAML), **Reset All** (replace all sections in the stored YAML with current recipe; recompute per section).

---

## 4. Compiler- and app-owned content

Some sections (or parts of sections) are created or maintained by the app/compiler rather than the user. We need to make that visible and control editability:

- **Requirement:** Either (a) allow users to edit but **highlight** compiler/app-created sections (or blocks), or (b) make those sections **non-editable** but still **displayed**.
- **LVGL:** Treated as compiler-owned (built from canvas + bindings). So either shown read-only in the dialog, or editable but clearly highlighted as compiler-generated (and likely overwritten on next compile).
- **Other compiler/app content:** Any section or block that is written by “Create Component,” binding sync, or other automation can be designated as compiler/app-owned. Same rule: either non-editable and displayed, or editable but highlighted so the user knows it may be replaced or patched by the app.
- **Concrete choice (for implementation later):** The design allows either policy: per section (or per block) we can store or derive a “source” (user | recipe | compiler/app) and then in the UI either make compiler/app content read-only or show it with a distinct style (e.g. background or badge “Generated by app”).

This gives a clear contract: user-owned content is normal edit; compiler/app-owned content is either locked or visually distinct so users understand it can change when the app runs.

---

## 5. How the app/compiler amends the stored YAML

When the app or compiler needs to change the stored YAML (e.g. Create Component, binding sync, building LVGL), the operation is one of:

- **Replace:** Overwrite the entire section with new content (e.g. compiler replaces the whole `lvgl` section).
- **Patch:** Update specific keys or blocks within an existing section (e.g. add `on_turn_on:` to an existing switch block for a binding).
- **Append:** Add a new block to a list section (e.g. Create Component adds a new `- platform: lvgl ...` entry under `switch:`).

Which operation is used **depends on the particular automation** (Create Component vs binding sync vs LVGL build, etc.). The design does not fix a single rule for all sections; each automation specifies replace, patch, or append as appropriate. If a section does not exist, the automation may have to create it (and then replace/patch/append).

- **List sections** (e.g. switch, sensor): Create Component typically **appends** a block; binding sync typically **patches** an existing block (e.g. by `id`).
- **Single-block sections** (e.g. logger, wifi): Automation might **replace** the whole section or **patch** specific keys.
- **LVGL:** Compiler **replaces** the whole section.

So the stored YAML is the base; each automation has a defined operation (replace / patch / append) and target (section or block within section).

---

## 6. Build pipeline (compile to final YAML)

When producing the final ESPHome YAML (e.g. for validate/flash or export):

1. **Start from** the stored YAML (user + recipe + any prior app amendments).
2. **Apply compiler-owned sections:** For each section (or block) owned by the compiler (e.g. LVGL), **replace** (or patch) with the compiler output. So the final document has compiler output for LVGL (and any other compiler-owned sections); the rest comes from the stored YAML.
3. **Apply any other app/compiler amendments** that are defined to run at build time (e.g. binding patches, script stubs). Each such amendment is defined as replace, patch, or append.
4. **Output** the resulting YAML.

So: stored YAML is the source of truth for user- and recipe-sourced content; compiler-owned content overwrites or patches at build time; other amendments follow their defined operation.

---

## 7. Summary table

| Topic | Decision |
|-------|----------|
| Source of truth | Single stored YAML per device (from recipe at creation, then edited/amended). |
| Reset | Restore section(s) to **current** hardware recipe (with substitutions). No “original at creation” snapshot. |
| Section states | Empty, Auto, Edited — derived from stored content vs current recipe. |
| Per-section actions | Save, Reset. |
| Dialog actions | Save All, Reset All. |
| Compiler/app-owned | Shown in dialog; either non-editable or editable but highlighted. LVGL = compiler-owned. |
| App amendments | Replace, patch, or append depending on the automation. Section created if missing where needed. |
| Build | Stored YAML → then apply compiler-owned sections (replace/patch) and any build-time amendments → final YAML. |
